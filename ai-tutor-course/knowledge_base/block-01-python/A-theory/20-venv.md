# Виртуальные окружения в Python

## Что такое виртуальное окружение

Виртуальное окружение (virtual environment, venv) — это изолированная копия Python со своим набором установленных библиотек. Каждый проект живёт в своём "пузыре" и не мешает другим проектам.

Проблема, которую решает venv: разные проекты требуют разные версии одних и тех же библиотек. Без изоляции они конфликтуют.

```bash
# Создание виртуального окружения
python -m venv venv

# Активация (Linux/Mac/Termux)
source venv/bin/activate

# Активация (Windows)
venv\Scripts\activate

# Деактивация
deactivate
```

## Аналогии виртуальных окружений

#### 🏠 Аналогия 1: Изолированные комнаты
Виртуальное окружение — как отдельная комната для каждого проекта. В одной комнате (проект А) стоит мебель в стиле "минимализм" (библиотека версии 1.0). В другой (проект Б) — "классика" (та же библиотека версии 2.0). Мебель не смешивается, каждый проект живёт в своём интерьере.

#### 🚗 Аналогия 2: Отдельный гараж для каждой машины
venv — как личный гараж. У каждой машины (проекта) свой гараж со своими запчастями (библиотеками). Запчасти от Toyota не валяются в гараже у BMW. Ты заходишь в нужный гараж (активируешь venv) и работаешь только с тем, что там лежит.

#### 🧪 Аналогия 3: Лабораторная песочница
venv — как стерильная лаборатория для каждого эксперимента. В одной лаборатории ты смешиваешь реактивы для опыта А, в другой — для опыта Б. Если что-то взорвётся в одной лаборатории, другая не пострадает.

#### 🎒 Аналогия 4: Личный шкафчик в школе
venv — как персональный шкафчик. У каждого ученика (проекта) свой шкафчик со своими учебниками (библиотеками). Ты не берёшь чужие учебники и не путаешь их со своими. Открыл свой шкафчик (активировал venv) — работаешь со своим.

#### 🦠 Аналогия 5: Карантинная зона
venv — как карантинный бокс. Если в одном проекте "заболела" библиотека (установилась кривая версия), инфекция не распространяется на другие проекты. Каждый проект изолирован.

## Зачем это нужно

### Проблема без venv

```bash
# Проект А требует requests==2.25.0
pip install requests==2.25.0

# Проект Б требует requests==2.31.0
pip install requests==2.31.0  # Перезаписал версию для проекта А!

# Теперь проект А сломался 💥
```

### Решение с venv

```bash
# Проект А — своё окружение
cd project_a
python -m venv venv
source venv/bin/activate
pip install requests==2.25.0  # Только для проекта А

# Проект Б — своё окружение
cd project_b
python -m venv venv
source venv/bin/activate
pip install requests==2.31.0  # Только для проекта Б, не мешает А
```

## requirements.txt

Файл `requirements.txt` — это список всех библиотек проекта. Позволяет воссоздать окружение на любом компьютере одной командой.

```bash
# Сохранить текущие библиотеки в файл
pip freeze > requirements.txt

# Установить все библиотеки из файла (на новом компьютере)
pip install -r requirements.txt
```

Пример `requirements.txt`:
```text
requests==2.31.0
fastapi==0.110.0
aiogram==3.4.0
sqlalchemy==2.0.25
```

## Полный рабочий процесс

```bash
# 1. Создаём папку проекта
mkdir my_project && cd my_project

# 2. Создаём виртуальное окружение
python -m venv venv

# 3. Активируем его
source venv/bin/activate
# В начале строки терминала появится (venv)

# 4. Устанавливаем нужные библиотеки
pip install requests

# 5. Работаем над проектом...

# 6. Сохраняем зависимости
pip freeze > requirements.txt

# 7. Когда закончили — деактивируем
deactivate
```

## .gitignore для venv

Папку `venv` **никогда не коммитят** в Git! Она содержит тысячи файлов и специфична для каждого компьютера. В `.gitignore` добавляют:

```text
venv/
__pycache__/
*.pyc
.env
```

Вместо папки `venv` в репозиторий кладут `requirements.txt` — по нему любой воссоздаст окружение.

## Типичные ошибки

### Ошибка 1: Забыли активировать venv
```bash
# ❌ Устанавливаешь в глобальный Python
pip install requests  # Попало в системный Python, не в проект!

# ✅ Сначала активируй
source venv/bin/activate
pip install requests  # Теперь в venv проекта
```

### Ошибка 2: Коммитят папку venv в Git
```bash
# ❌ venv содержит тысячи файлов, раздувает репозиторий
git add venv/  # Плохо!

# ✅ Добавь venv/ в .gitignore
echo "venv/" >> .gitignore
```

### Ошибка 3: Путают python и python3
```bash
# ✅ В активированном venv команда python указывает на venv
(venv) $ which python
# /path/to/project/venv/bin/python  # Это venv, а не системный!
```

## Итого

Виртуальное окружение изолирует библиотеки каждого проекта.
Создание: `python -m venv venv`.
Активация: `source venv/bin/activate` (Linux/Termux).
Деактивация: `deactivate`.
Зависимости сохраняют в `requirements.txt`: `pip freeze > requirements.txt`.
Восстановление: `pip install -r requirements.txt`.
Папку `venv/` добавляют в `.gitignore` — её не коммитят.
